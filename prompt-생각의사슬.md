# 1. Step-Back Prompting — "추상부터, 구체는 나중에"
## prompt
```
스탭백 사고(step-back prompting)
뒤로 한발짝 물러나서 생각해
focus on abstract principles before reasoning through details.
First derive high-level concepts and first principles before tackling specific details. 
```

## **원리**
구체적 문제에 바로 뛰어들면 지엽적 패턴에 갇힌다. 먼저 그 문제가 속한 상위 개념을 꺼내게 한 다음, 거기서 내려오는 방식.

## **적용 방법**
```
[나쁜 프롬프트]
"파이썬에서 리스트 컴프리헨션 쓰는 법 알려줘"

[Step-Back 적용]
Step 1: "리스트 컴프리헨션이 속하는 프로그래밍 패러다임은 무엇인가?"
Step 2: "그 패러다임의 핵심 원칙은?"
Step 3: "그 원칙이 파이썬 리스트 컴프리헨션에서 어떻게 구현되는가?"
```

## **효과**
단순 문법 암기가 아니라 함수형 프로그래밍의 원리에서 문법을 도출하게 되므로, 새로운 변형 문제에도 대응 가능한 답이 나온다. 암기 vs. 이해의 차이.

# 2. Contrastive Chain-of-Thought — "틀린 예시를 함께 보여줘"
##  prompt
```
대조적 연쇄 사고 (Contrastive Chain-of-Thought)
'인간은 긍정과 부정의 예시로부터 '학습한다' 에서 영감을 받음.
대조적 예시를 자동으로 생성하게 하는 방식
```

## **원리**
인간은 "이게 맞다"만큼 "이건 왜 틀렸나"에서 더 많이 배운다. 모델에게도 올바른 추론과 틀린 추론을 나란히 제시하면 경계면을 학습한다.

## **적용 방법**
```
[일반 CoT]
문제: 3 + 5 × 2 = ?
풀이: 3 + 10 = 13. 정답.

[Contrastive CoT]
문제: 3 + 5 × 2 = ?

[틀린 풀이]
3 + 5 = 8, 8 × 2 = 16. → 오류: 연산 순서 무시.

[맞는 풀이]
곱셈 우선: 5 × 2 = 10, 그 후 3 + 10 = 13. → 정답.
```

## **효과**
모델이 오류의 구조를 명시적으로 인식한다. 특히 **규칙 기반 추론, 논리 문제, 법률·의학적 판단** 등 오류 유형이 명확한 도메인에서 정확도가 유의미하게 올라간다. 단순히 "맞는 것"만 학습시킬 때보다 결정 경계가 선명해진다.

# 3. Analogical Prompting — "비슷한 문제부터 풀게 해"
##  prompt
```
유츄적사고 (Analogical Prompting)
어려운 수학 문제를 풀 때, 인간은 유사한 문제를 생각하고 풀이법을 생각한다
By drawing inspiration from human analogical reasoning. Inspired by our ability to draw connections between past experiences and current challenges.
```
## **원리**
인간이 낯선 문제를 만나면 먼저 유사한 경험을 탐색한다. 모델에게도 문제 풀기 전에 유사 사례를 스스로 생성하게 하면, 그 사례가 풀이 전략의 발판이 된다.

## **적용 방법**
```
[직접 질문]
"다음 수학 문제를 풀어라: ..."

[Analogical 적용]
Step 1: "이 문제와 구조가 유사한 문제를 2개 만들고 풀어라."
Step 2: "그 풀이 방식을 원래 문제에 적용하라."
```

실제 프롬프트 예:
```
원래 문제를 풀기 전에, 이와 유사한 더 단순한 문제를 직접 만들어서 먼저 풀어라.
그 풀이 전략을 원래 문제에 적용하라.
```

## **효과**
외부에서 예시를 주입하지 않아도 모델이 **자체적으로 few-shot 문맥을 생성**한다. 특히 수학, 코딩, 논증 구성처럼 전략 선택이 결과를 좌우하는 문제에서 효과가 크다. 예시를 직접 설계하기 어려울 때 유용.

4. Thread of Thought — "중간 기록을 남기면서 읽어"
# prompt
```
사고의흐름(Thread of Thought)
중요한 정보는 중간 중간 기록으로 남겨서 기록해.
인간이 선별하는 방식
복잡한 컨텍스트를 체계적으로 처리하고방대한 정보를 처리할 때 핵심 포인트를 요약 및 분석하여 관리
scratchpads for intermediate computation with language models
```

**원리**
방대한 문서를 한 번에 처리하면 앞 내용이 증발한다. 인간이 긴 책을 읽을 때 여백에 메모하듯, 모델이 컨텍스트를 처리하면서 핵심 포인트를 스스로 기록하게 한다.

**적용 방법**
```
[일반 처리]
"다음 50페이지 문서를 요약하라."

[Thread of Thought 적용]
"문서를 읽으면서, 각 섹션마다 핵심 주장을 한 줄로 기록하라.
모든 섹션을 처리한 후, 그 기록들을 종합해 최종 요약을 작성하라."
```

스크래치패드 방식:
```
[섹션 1 처리 중] → 핵심: "......"
[섹션 2 처리 중] → 핵심: "......"
[종합] → 위 핵심들 기반으로 최종 결론 도출
```

**효과**
긴 컨텍스트에서 발생하는 **중간 정보 손실**을 방지한다. 모델이 최종 답을 낼 때 앞서 추출한 기록을 참조하므로, 문서 전체를 균등하게 반영한 출력이 나온다. 계약서 분석, 논문 리뷰, 긴 회의록 처리에 특히 유효.
