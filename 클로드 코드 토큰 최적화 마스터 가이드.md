# **클로드 코드(Claude Code) 토큰 최적화 마스터 가이드: 52가지 전략 총정리**

## **\[프롤로그\] 토큰 최적화의 대원칙**

**연비 운전 이전에 주행 연습부터: 툴 숙련도의 중요성**

운전 면허를 막 딴 상태에서 연비 운전을 고민하는 것은 무의미합니다. 클로드 코드를 효율적으로 사용하기 위한 첫걸음은 역설적이게도 '최적화에 얽매이지 않고 마음껏 써보는 것'입니다. 툴이 제공하는 다양한 기능과 응답 패턴을 내 몸의 일부처럼 자연스럽게 다룰 수 있을 때까지 충분히 사용해 보는 숙련도 확보가 선행되어야 합니다. 도구의 특성을 온전히 이해한 후 적용하는 최적화 전략이야말로 진정한 생산성 혁명으로 이어집니다.

**점진적 도입 전략: 필수 3대장부터 시작하라**

이 가이드에 담긴 52가지의 팁을 한꺼번에 모두 적용하려고 하면 시스템 운영에 오히려 부담이 되고 역효과가 날 수 있습니다. 모든 것을 한 번에 통제하려 하지 말고, 가장 파급력이 큰 필수 요소부터 단계적으로 도입하십시오. **작업 전후 주기적인 초기화(/clear)**, **무의미한 탐색을 막는 .claudeignore 파일 생성**, 그리고 **안 쓰는 툴 정리**와 같은 즉각적이고 효과가 큰 팁부터 내재화하는 것을 강력히 추천합니다.

## **\[1단계\] 즉각적 지혈 (Tier 1: Immediate Action)**

*가장 적은 노력으로 즉시 토큰 누수를 막는 필수 세션 위생 습관입니다.*

### **1.1 세션 및 컨텍스트 초기화의 생활화**

* **작업 전후 /clear 리셋으로 기하급수적 토큰 증가 차단:** 클로드는 새로운 메시지를 보낼 때마다 과거의 전체 대화 기록을 처음부터 다시 읽어 들입니다. 즉, 대화가 길어질수록 소모되는 토큰은 선형적이 아닌 기하급수적으로 폭증합니다. 작업 단위가 끝날 때마다 제때 /clear를 입력해 컨텍스트를 비워주는 것만으로도 막대한 토큰 낭비를 즉시 차단할 수 있습니다.  
* **한 세션, 단일 논리 작업 원칙:** 하나의 세션 창에서 버그 수정, 기능 추가, 리팩토링 등을 동시에 진행하지 마십시오. 서로 다른 작업의 컨텍스트가 뒤섞이면서 AI의 연산량과 토큰 사용량이 급증합니다. 한 세션에서는 오직 하나의 논리적인 작업만 집중력 있게 처리하는 것이 원칙입니다.  
* **5분 캐시 만료 전 압축(/compact) 및 초기화 습관:** 클로드 내부의 프롬프트 캐시는 약 5분 뒤에 만료됩니다. 자리를 오래 비웠다가 돌아와서 작업을 재개하면, 캐시가 날아가 전체 컨텍스트가 처음부터 재처리되며 토큰이 확 오를 수 있습니다. 자리를 뜰 때는 미리 /compact 명령어로 핵심만 압축해두거나 과감히 /clear를 실행하십시오.

### **1.2 타겟팅 지시와 최소주의 프롬프팅**

* **파일 전체(@) 참조 지양 및 특정 줄 타겟팅:** README.md나 수천 줄짜리 코드를 파일째로 태깅하여 "개선해 줘"라고 막연히 명령하면, 수정이 필요 없는 내용까지 모두 컨텍스트에 담겨 낭비됩니다. "10번째 줄의 특정 함수를 수정해 줘"처럼 작업 구간을 콕 집어서 타겟팅하는 습관을 들여야 합니다.  
* **꼭 필요한 내용만 붙여넣기 및 간단한 명령 번들링:** 오류 코드나 참조 문서를 복사할 때는 핵심 텍스트만 선별하여 최소한으로 붙여넣으십시오. 또한 코드 요약, 핵심 포인트 추출, 제목 추천 같은 가벼운 명령들은 매번 따로 보내지 말고 한 번의 메시지로 묶어서(번들링) 처리하는 것이 효율적입니다.

### **1.3 모델 및 탐색 기본 설정 방어**

* **.claudeignore 파일로 무의미한 탐색 원천 차단:** 클로드가 프로젝트 구조를 파악할 때 node\_modules, .next, build 등 수십 기가바이트에 달하는 불필요한 디렉토리까지 모두 스캔하려 들 수 있습니다. 프로젝트 루트에 반드시 .claudeignore 파일을 생성하여 탐색 금지 대상을 명시하십시오.  
* **기본 모델 Sonnet 고정 및 작업별 유연한 분배:** 평상시 디폴트 모델은 가성비와 성능의 밸런스가 가장 뛰어난 Sonnet으로 고정되어 있는지 확인하십시오 (종종 가장 비싼 Opus로 설정되어 토큰이 녹아내리는 경우가 있습니다). 단순한 문답이나 데이터 추출은 저렴한 Haiku로, 깊은 아키텍처 설계나 디버깅 시에만 Opus를 투입하는 전략적 모델 배분이 필요합니다.  
* **자동화 모드(Shift+Tab) 진입 시 무한 루프 감시:** Shift+Tab을 활용해 자동 수락 모드로 전환하고 자리를 비우는 것은 위험할 수 있습니다. AI가 문제 해결의 방향을 잘못 잡아 끝없는 무한 루프에 빠진 채로 토큰을 낭비할 수 있으므로, 실행 초반에는 의도한 올바른 방향으로 나아가고 있는지 반드시 모니터링해야 합니다.

## **\[2단계\] 구조적 다이어트 (Tier 2: Structural Optimization)**

*매 메시지마다 기본적으로 발생하는 '베이스 코스트(Base Cost)'를 시스템적으로 낮추는 작업 환경 및 파일 구조 재설계 단계입니다.*

### **2.1 claude.md의 극한 다이어트**

* **매 메시지마다 로드되는 claude.md 200줄 이하 유지:** 클로드는 세션 시작뿐만 아니라 매 메시지를 보낼 때마다 claude.md 파일을 읽습니다. 따라서 이 파일은 서술형 문장을 과감히 덜어내고 리스트나 테이블 형태로 구조화하여 200줄 이하로 가볍게 유지해야 합니다.  
* **토큰 낭비를 막는 핵심 규칙 및 컨벤션 명시:** 이미 팀 내에서 합의된 코딩 컨벤션이나 아키텍처 규칙을 미리 적어두면 반복적인 프롬프트 타이핑을 줄일 수 있습니다. 특히 "이미 읽은 파일은 재확인하지 않는다", "불필요한 도구 호출은 금지한다"와 같은 토큰 절약 규칙 5가지를 명시하면 AI 스스로 낭비를 통제하게 됩니다.  
* **분산 메모리 구조 (스킬 파일 분리):** claude.md에 모든 것을 몰아넣지 마십시오. API 가이드, DB 구조 등 세부적이고 방대한 지식은 하위 폴더의 '스킬(Skill) 파일'로 분리하고, claude.md는 이를 가리키는 '인덱스(목차)' 역할만 수행하도록 설계해야 합니다.

### **2.2 MCP(Model Context Protocol) 오케스트레이션**

* **4단계 MCP 다이어트 및 미사용 툴 퇴출:** 연결된 MCP 서버는 아무 명령을 내리지 않아도 매번 도구 정의를 로드하며 수만 토큰을 소모합니다. 2주 이상 사용하지 않은 툴은 과감히 퇴출하십시오. (1) 연동 목록 확인 → (2) 장황한 설명 체크 → (3) 예상 토큰 추정 → (4) 오버헤드 식별 및 제거의 4단계를 거쳐 도구를 정밀하게 솎아내야 합니다.  
* **글로벌 레벨 vs 프로젝트 레벨 분리:** 모든 MCP를 전역(Global)에 연동하면 가벼운 프로젝트에서도 무거운 도구들이 로드됩니다. 파일 시스템 도구 정도만 전역에 두고, Supabase나 Playwright 같은 무거운 도구는 해당 프로젝트 레벨에만 독립적으로 연동하십시오.  
* **툴 서치(Tool Search) 옵션 활성화:** MCP 설정에서 툴 서치 옵션을 true로 켜두면, 수천 개의 도구를 매번 전부 읽어 들이는 대신 실시간으로 필요한 도구만 검색하여 불러옵니다. 이를 통해 도구 관련 토큰을 최대 85%까지 절감할 수 있습니다.

### **2.3 컨텍스트 분리 및 보호 전략**

* **플랜 모드(Plan Mode)로 실행 전 선검토:** 코드를 작성하기 전 Shift \+ Tab을 두 번 눌러 플랜 모드에 진입하십시오. AI가 제안하는 수정 파일과 작업 순서를 미리 검토하고 승인하는 과정을 거치면, 잘못된 방향으로 코드를 짰다가 다시 지우며 발생하는 2배의 토큰 낭비를 원천 봉쇄할 수 있습니다.  
* **대규모 작업 핸드오버 (progress.md):** 컨텍스트가 꽉 차서 어쩔 수 없이 초기화를 해야 할 때는 "지금까지의 완료 상황과 다음 단계를 progress.md로 정리해 줘"라고 명령하십시오. 리셋 후 해당 파일만 참조하게 하면 컨텍스트 유실 없이 거대한 작업을 안정적으로 이어갈 수 있습니다.  
* **멀티 터미널 전략 활용:** 하나의 터미널 창에서 피처 개발, 리팩토링, 버그 수정을 번갈아 진행하면 서로 다른 작업의 자녀 컨텍스트가 뒤섞여 토큰이 급증합니다. 터미널 창을 여러 개 띄워 작업별로 독립적인 세션을 유지하십시오.

## **\[3단계\] 시스템 자동화 제어 (Tier 3: Advanced Automation)**

*인간의 개입 없이도 토큰 낭비를 백그라운드에서 자동 차단하는 프로페셔널 환경 세팅입니다.*

### **3.1 훅(Hook)을 통한 전처리 및 차단**

* **read-once 훅으로 중복 재읽기 완전 차단:** 클로드는 같은 세션 내에서 이미 읽은 파일을 확인차 계속 다시 읽으려는 경향이 있습니다. 오픈소스인 read-once 훅을 설치하면 이미 읽은 파일의 추적이 가능해져 불필요한 재읽기를 차단하거나, 변경된 줄만 읽어오는 '디프(Diff) 모드'를 강제할 수 있습니다.  
* **명령어 출력 훅으로 에러 로그만 선별:** 전체 테스트 코드나 빌드를 실행할 때 발생하는 수백 개의 '성공' 로그가 컨텍스트로 쏟아지는 것을 막아야 합니다. 출력 훅을 걸어 failed나 error가 포함된 핵심 줄만 전처리하여 클로드에게 전달되도록 세팅하십시오.

### **3.2 코드베이스 사전 인덱싱 (QMD)**

* **탐색 비효율 90% 제거:** 클로드가 프로젝트 내 특정 코드를 찾을 때 전체 파일을 열고 키워드를 검색(Grep)하는 과정은 막대한 토큰을 소모합니다. 프로젝트를 미리 색인해 주는 QMD(코드베이스 사전 인덱싱)를 도입하면, 검색 엔진처럼 원하는 코드로 바로 직행할 수 있어 탐색 토큰을 극적으로 아낄 수 있습니다.

### **3.3 환경 변수(Env Vars)를 활용한 극한의 제어**

* **DISABLE\_NON\_ESSENTIAL\_MODEL\_CALLS=1:** 핵심 워크플로우에 영향을 주지 않으면서 눈에 보이지 않게 뒤에서 소모되는 백그라운드 모델 호출을 비활성화합니다.  
* **DISABLE\_PROMPT\_CACHING=0:** 반복되는 프롬프트나 수정되지 않은 claude.md를 캐싱하여 매 메시지마다 토큰을 획기적으로 줄여주는 캐싱 기능이 절대 꺼지지 않도록 강제합니다.  
* **DISABLE\_COST\_WARNINGS=1:** 외부 도구를 통해 이미 모니터링을 하고 있다면, 시스템이 자체적으로 보내는 비용 경고 메시지 알림조차 토큰을 소모하므로 이를 꺼두는 것이 좋습니다.

### **3.4 서브 에이전트 및 오픈소스 MCP 마이크로 매니징**

* **서브 에이전트는 Haiku 전용으로 단발성 투입:** 서브 에이전트는 깨어날 때마다 자신만의 전체 컨텍스트를 새로 로드하므로 일반 세션보다 토큰을 최대 10배나 많이 씁니다. 따라서 복잡한 작업보다는 단순 리서치나 탐색 같은 단발성 작업에만 저렴한 Haiku 모델을 지정하여 위임하십시오.  
* **오픈소스 MCP 장황한 설명 축약:** 연동된 오픈소스 MCP의 도구 설명(Description)이 사람이 읽기 좋게 너무 길게 쓰여 있다면, 이를 직접 포크(Fork)하여 핵심 의미만 남기고 분량을 쳐내야 합니다. 또한, 특정 서버에서 지원하는 필터링 플래그를 활용해 30개 도구 중 실제 사용하는 8개만 노출하도록 강제 제한하십시오.

## **\[부록\] 모니터링 및 복구 마스터 클래스 (Master's Toolbox)**

* **명령어 3총사 정밀 타격:**  
  * /context: 시스템 프롬프트, 대화 등 어느 컴포넌에서 토큰이 새고 있는지 점검합니다.  
  * /cost: 현재 세션의 누적 사용량과 예상 비용을 실시간으로 확인합니다.  
  * /usage: 주간 단위의 사용량까지 파악하여 예산 페이스를 조절합니다.  
* **세션 보관과 복구 (/rename & /resume):** 컨텍스트가 꽉 차서 초기화를 해야 하지만 나중에 히스토리가 필요할 것 같다면, /clear 이전에 /rename으로 현재 세션에 이름을 붙여 보관하십시오. 언제든 /resume 명령어로 해당 상태를 복구해 이어서 작업할 수 있습니다.  
* **실시간 대시보드 및 상태 표시줄(Status Line) 세팅:** 터미널 하단에 컨텍스트 사용률을 실시간으로 띄워 수동 압축(/compact) 타이밍을 직관적으로 잡으십시오. 또한 npm 도구인 CCCG를 설치하면 일/월 단위 사용량을 웹 대시보드처럼 띄워놓고 낭비 구간을 추적할 수 있습니다.  
* **Overage(초과 과금) 기능 활용:** 요금제 한도에 도달했을 때 중요한 프로젝트의 흐름이 끊기는 것이 치명적이라면, Overage 기능을 켜두십시오. 초과분만 API 요금으로 자동 과금되며 작업 중단을 막아주는 최후의 안전장치가 됩니다.
